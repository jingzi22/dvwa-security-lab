# 漏洞名称：SQL 注入漏洞

测试时间：2026.03  
测试环境：DVWA  
测试人员：赵磊  

---

## 一、漏洞等级

高危

---

## 二、漏洞描述

在用户输入参数未进行有效过滤或参数化处理的情况下，
攻击者可构造恶意 SQL 语句，改变原有查询逻辑，
从而获取数据库中的敏感信息。

---

## 三、漏洞复现步骤

1. 访问 DVWA SQL Injection 模块页面
2. 在输入框中输入以下 payload：

```
' OR 1=1 --
```

3. 提交请求
4. 页面返回所有用户数据

---

## 四、请求响应证据

### 1️⃣ 原始请求

（此处插入 Burp 抓包截图）

示例：

```
GET /vulnerabilities/sqli/?id=1' OR 1=1 -- &Submit=Submit HTTP/1.1
```

---

### 2️⃣ 响应结果

（此处插入页面返回数据截图）

页面显示多个用户记录，说明查询条件被绕过。

---

## 五、漏洞原理分析

应用程序直接将用户输入拼接到 SQL 语句中，例如：

```sql
SELECT * FROM users WHERE id = '$id';
```

当输入：

```
' OR 1=1 --
```

实际执行语句变为：

```sql
SELECT * FROM users WHERE id = '' OR 1=1 -- ';
```

其中：

- `OR 1=1` 恒为真
- `--` 注释后续语句

导致数据库返回全部数据。

---

## 六、风险影响

- 数据库敏感信息泄露
- 管理员账号被枚举
- 数据被篡改或删除
- 进一步系统渗透

---

## 七、修复建议

1. 使用参数化查询（Prepared Statement）
2. 禁止字符串拼接 SQL 语句
3. 启用输入校验与过滤
4. 部署 WAF 进行防护
5. 最小权限原则配置数据库账户

---

## 八、复测标准

修复后再次使用以下 payload：

```
' OR 1=1 --
```

页面应返回：

- 查询错误提示
或
- 仅返回合法数据

不得返回全部数据。

---

## 九、总结

本漏洞属于典型输入未过滤导致的 SQL 注入问题，
风险等级高，应优先修复。
